
Task is to design and implement an open-source Windows tool called **Losshound** from scratch, end to end.

# Product goal

Build a lightweight Windows network diagnosis tool that continuously monitors connectivity and determines whether current failures are most likely caused by:

1. LAN / local network
2. Router / gateway path
3. ISP / WAN connection
4. DNS failure
5. Upstream / internet routing issue
6. Route instability / path changes

The tool must be useful for normal users but technically credible for power users.

# Core concept

Losshound should continuously compare:

- Router/default gateway ping
- Public IP ping
- DNS resolution health
- Route/traceroute changes over time

Then it should infer the most likely fault domain and present a clear status like:

- "LAN issue likely"
- "ISP / WAN instability likely"
- "DNS issue likely"
- "Upstream route issue likely"
- "Connection healthy"
- "Intermittent packet loss detected"

# Product requirements

## Functional requirements

Losshound must:

- Detect the default gateway automatically
- Ping the default gateway on a schedule
- Ping at least one public IP on a schedule, ideally configurable defaults such as:
  - 1.1.1.1
  - 8.8.8.8
- Perform DNS resolution tests on a schedule, for example:
  - resolve google.com
  - resolve chatgpt.com
- Periodically run route checks and detect route changes
- Track:
  - packet loss
  - average latency
  - jitter estimate
  - timeout bursts
  - route differences over time
- Keep a rolling time window of recent observations
- Infer likely cause of failure using explicit decision logic
- Show current status and recent history
- Export logs or a diagnostic report

## Inference logic requirements

Implement a transparent rule-based diagnosis engine. Do not use ML for the core diagnosis.

The logic should distinguish cases like:

### LAN issue likely
Examples:
- gateway ping failing or high loss
- public IP ping failing too
- DNS failing too

### ISP / WAN issue likely
Examples:
- gateway stable
- public IP ping failing or showing high loss
- DNS also failing because internet path is broken

### DNS issue likely
Examples:
- gateway stable
- public IP ping stable
- DNS resolution failing

### Upstream route issue likely
Examples:
- gateway stable
- public IP reachable intermittently
- route path changing often
- route destination reachable but with instability outside local network

### Healthy
Examples:
- low latency
- near-zero packet loss
- stable DNS
- no suspicious route changes

### Intermittent instability
Examples:
- short bursts of loss or timeouts
- route remains reachable
- destination loss appears only periodically

The diagnosis engine must be clearly implemented in a dedicated module with readable comments and easily adjustable thresholds.

# Technical implementation notes

## Gateway detection
- Detect default gateway on Windows using reliable Python or subprocess methods

## Ping implementation
- Prefer reliable subprocess-based ping parsing for Windows compatibility
- Handle timeouts, packet loss, average RTT, min/max RTT
- Avoid requiring raw sockets or admin privileges

## DNS checks
- Use Python socket or dnspython if needed
- Measure success/failure and resolution time
- Make resolver targets configurable if practical

## Route monitoring
- Use `tracert` on Windows
- Parse output into hop lists
- Compare route snapshots over time
- Detect meaningful route changes, not just cosmetic noise
- Gracefully handle hops that do not respond

## Scheduling
- Background worker or timer-based approach
- Do not block the GUI
- Ensure the app remains responsive

## Persistence
- Store recent history locally in JSON or SQLite
- Save enough information for useful troubleshooting
- Rotate or cap logs/history to avoid bloat

## Reporting
The app should be able to export a plain-language report such as:
- timestamp
- gateway status
- public IP status
- DNS status
- route snapshot
- inferred diagnosis
- short explanation

# Decision engine detail

Implement explicit thresholds in configuration. For example:

- gateway_loss_threshold
- public_loss_threshold
- dns_failure_threshold
- latency_warning_ms
- jitter_warning_ms
- route_change_sensitivity
- timeout_burst_threshold

Then build human-readable explanations, for example:
- "Gateway is stable, but public IP tests are failing. This suggests WAN/ISP instability."
- "Public IP remains reachable while DNS resolution repeatedly fails. DNS issue likely."
- "Route path changed 3 times in the last 10 minutes while gateway remained stable. Upstream route instability likely."
